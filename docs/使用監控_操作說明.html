📊 使用監控 — 完整操作說明

自動記錄每位使用者的 API 呼叫行為，提供模組使用統計、用戶活躍度、效能分析與錯誤率監控。

v1.69.26 新功能 system_admin 專屬 無需額外設定 即時更新

🔍 什麼是使用監控

使用監控是一套內建於 Portal 後端的行為追蹤系統。每當使用者在任何模組執行操作（開啟列表、查詢資料、儲存記錄等），系統會自動在背景記錄一筆日誌，包含：

記錄誰
用戶 Email
從登入 Token 自動解出
記錄哪個模組
模組名稱
從 API 路徑自動解析
記錄做了什麼
讀取 / 寫入
GET = 瀏覽；其他 = 操作
記錄多快
毫秒計時
每筆紀錄含回應時間
ℹ️
所有記錄均為自動進行，不需要使用者或管理員手動操作。系統從後端 API 層攔截，不影響任何現有功能的速度或行為。

🔐 誰能看到

使用監控頁面僅限 系統管理員（system_admin） 角色存取，一般用戶與據點管理員無法看到此頁面及相關 API。

⚠️
使用監控統計包含所有用戶的行為資料。管理員應注意資料使用符合內部規範，僅作為系統維護與效能優化之用途。

統計區間切換

右上角下拉選單可切換 1 / 7 / 14 / 30 天。切換後所有 TAB 數據同步更新。點擊 ↻ 重新整理 圖示可手動重新載入最新資料。

📋 TAB：模組使用排行

顯示各模組在統計區間內的使用次數，依總次數由高至低排序。

欄位說明
模組API 路徑解析出的模組名稱，例如 room-maintenance、security-patrol
總請求所有方法合計
瀏覽（GET）查詢類請求，代表「有人打開這個頁面/查詢了資料」
操作（寫入）POST / PATCH / DELETE，代表「有人實際修改了資料」
💡
使用技巧：瀏覽數遠大於操作數的模組，通常是「只看不動」的查詢頁面。若某模組瀏覽數突然暴增，可能是有用戶在大量下載或頻繁查詢。

👥 TAB：用戶活躍度

顯示統計區間內各用戶的操作次數排行，預設顯示前 20 名。

欄位說明
Email用戶帳號（從登入 Token 記錄）
請求數統計區間內的總操作次數
最常用模組該用戶使用最頻繁的模組名稱
最後活動最後一次 API 呼叫的時間（台灣時間）
ℹ️
表格可點擊「請求數」欄位標題進行升降序排列。若某用戶長期不活躍，可考慮停用帳號以確保資安。

⚡ TAB：回應時間

分析各模組的 API 效能，找出需要優化的慢速端點。

欄位說明
平均（ms）所有請求的平均回應時間
P95（ms）第 95 百分位數。95% 的請求都在此時間內完成，剩下 5% 更慢。比平均值更能反映「慢的時候有多慢」
最慢（ms）統計區間內最慢的單次回應

顏色代碼

綠色 < 200ms 良好，用戶感受流暢 橘色 < 800ms 偶爾感受到延遲 紅色 ≥ 800ms 明顯卡頓，建議排查

💡
P95 是最重要的指標。即使平均值正常，若 P95 過高（例如 2000ms），代表每 20 次操作中就有 1 次體驗很差，用戶可能已有感受。

📉 TAB：DAU 趨勢

顯示過去 N 天每日不重複的活躍用戶數折線圖，幫助了解系統整體使用趨勢。

ℹ️
DAU 定義：當天曾呼叫任一業務 API（非 login/logout）的不重複 user_id 數量。
若某天 DAU 異常低（例如假日、節慶），屬正常現象。若工作日持續低迷，建議了解原因。

圖表操作

滑鼠移到圖表上可查看各日期詳細數值。切換右上角統計區間可看 7 天 / 14 天 / 30 天趨勢。

📊 TAB：請求量時間軸

以每小時為單位顯示 API 請求量長條圖，可看出系統使用的高峰與低谷時段。

用途說明
找出使用高峰通常在 08:30–11:00 和 14:00–16:00 出現峰值
排程維護時間選擇請求量最低的時段進行系統重啟或更新
識別異常流量非工作時間出現高峰，可能是自動化腳本或異常存取

🚨 TAB：錯誤率

顯示各模組的 HTTP 錯誤（4xx / 5xx）統計，依錯誤數由高至低排序。

欄位說明
4xx用戶端錯誤（403 權限不足、404 找不到、429 被 Rate Limit 等）
5xx伺服器錯誤，代表後端程式或資料庫異常，需要立即排查
錯誤率(4xx + 5xx) ÷ 總請求 × 100%
⚠️
任何模組出現 5xx 錯誤，應立即查看後端 logs/ 目錄的對應 log 檔案。4xx 若持續偏高（> 5%），可能是某用戶反覆碰到權限問題，建議檢視角色設定。

🚀 如何進入使用監控

1
以 系統管理員（system_admin） 帳號登入 Portal
2
點擊左側選單「系統設定」群組展開
3
點擊「使用監控」進入頁面
4
右上角選擇統計區間（1 / 7 / 14 / 30 天），點擊各 TAB 查看不同維度的分析

💼 常見使用情境

情境 1：了解哪個模組最常被使用

進入「模組使用排行」TAB，選擇 30 天統計區間，查看總請求數排行。排名高的模組代表使用者最依賴，優化時應優先考慮這些模組的效能。

情境 2：哪些員工沒有在用系統

進入「用戶活躍度」TAB，選擇 30 天統計區間。若某帳號完全不出現在清單，代表該用戶過去 30 天沒有操作任何模組，可考慮確認帳號狀態。

情境 3：系統感覺變慢，找出哪個模組有問題

進入「回應時間」TAB，以 P95 欄位排序（點擊欄位標題），找出 P95 最高的模組。若超過 1000ms，可進一步查看後端 log 或資料庫查詢是否有效率問題。

情境 4：安排系統維護時間

進入「請求量時間軸」TAB，選擇 7–14 天，觀察哪個時段請求量最低（通常是凌晨 01:00–06:00）。選擇此時段進行重啟或更新，對用戶影響最小。

情境 5：確認新功能是否有人在用

新模組上線後，進入「模組使用排行」TAB，查看該模組的請求數。若上線後一週請求數為 0，可能是權限設定未開放或選單未顯示，需要排查。

❓ 常見問題

使用監控是否會影響系統效能？
不會。所有日誌寫入均採用背景非同步處理（fire-and-forget），不會增加 API 的回應時間。即使寫入 DB 發生錯誤，也只會在系統 log 留下記錄，不影響使用者操作。
登入、登出有被記錄嗎？
登入與登出事件記錄在 audit_logs 資料表（含 IP 位址），這是更早期就有的資安功能。使用監控的 api_access_logs 只記錄模組操作，兩者分開管理，各司其職。
DAU 只算進來看頁面，還是要實際操作才算？
兩者都算。只要用戶開啟任何模組頁面（前端會呼叫 GET API 取得資料），就會被計入 DAU。單純登入後停留在 Dashboard 但不點任何模組，不計入 DAU。
資料會保存多久？會自動清除嗎？
目前版本不自動清除，資料一直累積在 SQLite 資料庫的 api_access_logs 資料表。若資料量過大，可由系統管理員手動執行 SQL 清理（例如保留最近 6 個月）。未來版本可能加入自動保留期設定。
定時自動同步（APScheduler）的 API 呼叫有被記錄嗎？
沒有。APScheduler 排程任務直接在伺服器內部呼叫 service 函數，不走 HTTP 請求，因此不會經過 Middleware，也不會出現在使用監控記錄中。這是設計上的刻意排除，確保統計數字代表真實的「人為操作」。
統計頁面右上角的時間區間變更後，所有 TAB 都會更新嗎？
是的。切換統計區間後，所有 TAB（模組排行、用戶活躍度、回應時間、DAU、時間軸、錯誤率）都會重新拉取對應時段的資料。
為什麼有些 API 路徑沒有出現在模組排行？
以下路徑被系統刻意排除，不會出現在記錄中：
  • /api/v1/auth/ — 登入、登出、取得個人資訊
  • /api/docs、/api/redoc — API 文件頁面
  • /assets/、/favicon — 前端靜態資源
  • GET /api/v1/auth/me — 頁面重整時的身份驗證查詢（高頻、無意義）

🗄️ 資料表說明

使用監控使用一張新的資料表 api_access_logs，與原有的 audit_logs 分開存放。

api_access_logs — 欄位說明

欄位型別說明
idUUID主鍵，自動產生
user_idString(36)從 JWT Token 解出的用戶 UUID；未登入時為 NULL
user_emailString(100)用戶 Email（冗餘存放，避免 JOIN）
moduleString(80)從 API 路徑解析的模組名稱
methodString(10)HTTP 方法（GET / POST / PATCH / DELETE）
pathString(250)完整 API 路徑
status_codeIntegerHTTP 回應狀態碼
response_msInteger回應時間（毫秒）
ip_addressString(45)用戶 IP（支援 IPv6，優先讀取 X-Forwarded-For）
created_atDateTime記錄時間（台灣時間 UTC+8）

模組名稱解析規則

API 路徑解析結果
/api/v1/room-maintenance/…room-maintenance
/api/v1/mall/b4f-inspection/…b4f-inspection
/api/v1/security/patrol/…patrol
/api/v1/settings/users/…settings/users
/api/v1/dashboard/…dashboard

🚫 排除清單（不記錄的路徑）

路徑前綴原因
/api/v1/auth/登入 / 登出由 audit_logs 負責，避免重複記錄
/api/docs、/api/redoc、/api/openapi.jsonAPI 文件存取，非業務操作
/assets/、/favicon前端靜態資源
GET /api/v1/auth/me（精確排除）每次頁面重整都會呼叫，頻率極高、無分析意義
💡
若未來需要新增排除項目（例如 health check API），可修改 backend/app/middleware/audit_middleware.py 中的 _EXCLUDE_PREFIXES 或 _EXCLUDE_EXACT 常數。
