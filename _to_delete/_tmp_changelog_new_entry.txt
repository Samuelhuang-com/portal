## [1.80.53] - 2026-07-17

### Changed — 週期採購「請購單流程」大改版：拿掉送出／核准，改成「關閉／重新開啟」

- **背景**：Samuel 提出新規格：(1) 請購單不需要「送出」與「核准」；(2) 「當期」= `YYYY-MM` + 3 位流水編號，`period_label` 改由系統在建立當下直接蓋章為「YYYY-MM」，使用者不能修改，自己申請的請購單當月可編輯；(3) 增加「關閉」權限功能，當月請購單關閉後無法新增/編輯，關閉可以「全部關閉」或「勾選請購單」。先出具評估規劃、Samuel 確認 5 個決策點後才開始 coding
- **決策點確認**：獨立新增 `cycle_purchase_close` 權限（不與既有權限合併）；`request_no` 保留 `PR-` 前綴（`PR-2026-07-001`）；需要「重新開啟已關閉的請購單」；對舊資料做一次性轉換（`status='approved'` 視為 `is_closed=True`）；Dashboard「待簽核」卡片改成「這個月還有 N 張請購單沒關閉」
- **`backend/app/models/cycle_purchase_request.py`**：`request_no` 格式改為 `PR-YYYY-MM-NNN`（3 位流水號每月重新起算，舊資料 `PR-YYYYMM-NNNN` 格式不回溯改寫）；`period_label` 改為系統蓋章欄位；新增 `is_closed`／`closed_by_user_id`／`closed_by_name`／`closed_at`／`close_batch_no`／`reopened_by_user_id`／`reopened_by_name`／`reopened_at`；`status`／`submitted_*`／`approved_*`／`reject_reason` 降級為改版前歷史殘留欄位（新資料固定寫 `draft`，不再驅動任何邏輯）
- **`backend/app/services/cycle_purchase_request_service.py`**：`_next_request_no()` 改新格式；新增 `_current_period_label()`（系統蓋章用）、`_check_editable()`（同時檢查「沒被關閉」＋「還是當月」）；`update_request()`／`add_request_item()`／`update_request_item()`／`delete_request_item()` 一律改用 `_check_editable()`；整個移除 `submit_request()`／`approve_request()`／`reject_request()`；新增 `list_open_requests_for_close()`、`close_requests()`、`close_all_requests()`、`reopen_requests()`；`get_dashboard_todos()` 的 `is_approver`／`pending_approval` 改成 `is_closer`／`pending_close`（本月還沒關閉的請購單）
- **`backend/app/services/cycle_purchase_summary_service.py`**：`list_eligible_requests()`／`generate_summary_from_requests()` 的判斷條件從 `status == approved` 改成 `is_closed == True`；月份篩選從「`approved_at` 換算年月」改成直接比對 `period_label`；`_period_label_from_requests()` 改成直接讀 `period_label`（不再用 `approved_at.strftime`）
- **Schema／Router**：`cycle_purchase_request.py`／`cycle_purchase_summary.py` schemas 新增關閉／重新開啟相關欄位與 payload（`CloseRequestsPayload`／`CloseAllRequestsPayload`／`ReopenRequestsPayload`），移除 `RequestRejectPayload`；`cycle_purchase_requests.py` router 移除 `/submit`／`/approve`／`/reject`，新增 `/requests/open-for-close`（`GET`）、`/requests/close`、`/requests/close-all`、`/requests/reopen`；`role_permissions.py` 移除 `cycle_purchase_approve`、新增 `cycle_purchase_close`
- **前端**：`types/cyclePurchase.ts`／`api/cyclePurchase.ts` 同步新增關閉／重新開啟型別與 API（移除 `submitRequest`／`approveRequest`／`rejectRequest`）；`Requests/index.tsx` 新增「關閉請購單」視窗（週期＋公司＋月份篩選 → 勾選清單 →「全部關閉」／「關閉勾選」二選一）；`Requests/Detail.tsx` 拿掉送出/簽核按鈕與退回原因 Modal，改成「關閉此請購單」／「重新開啟」，並依 `is_closed`＋當月判斷是否可編輯；`Dashboard.tsx`「待簽核」卡片改成「本月待關閉」（依 `pending_close`／`pending_close_count`）；`Summary/index.tsx` 的「產生彙整」勾選清單欄位、提示文字同步從「核准人／核准時間」改成「關閉人／關閉時間」
- **一次性資料轉換**：`apply_cycle_purchase_summary_migration.py` 新增 `convert_legacy_approved_to_closed()`，把舊資料 `status='approved'` 轉成 `is_closed=True`（`closed_*` 從既有 `approved_*` 回填），用 `close_batch_no IS NULL` 判斷「還沒轉換過」，重複執行不會覆蓋使用者之後手動關閉／重新開啟的結果；另外新增獨立 SQL 檔 `add_cycle_purchase_request_close_columns.sql`（DDL 參考用，建議仍以 `.py`／`.bat` 為主）
- **驗證**：`py_compile` 全數通過；前端改動用 `tsc --noEmit` 逐檔確認無 `TS1xxx`/`TS17xx` 語法錯誤；所有觸及檔案改用「先在裝置端 `sha256sum` 驗證、再 `mv -f` 原地覆蓋、覆蓋後再 `sha256sum` 一次」的流程寫回，避免此類大改版常見的裝置端截斷風險
- **已知限制**：本次沒有再對 `ragic-cycle-purchase/docs/` 底下的規劃文件（`週期採購_Portal規劃評估_v1.1.md`、`欄位規格.md`）補上對應章節，留待下次一併處理
